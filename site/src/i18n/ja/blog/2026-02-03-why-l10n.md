複数の言語でソフトウェアをリリースした経験があれば、その手順はご存知でしょう。ローカリゼーションプラットフォームを選び、リポジトリに接続し、残りの時間は同期の管理に費やします。コンテンツが送られ、翻訳が返ってきますが、その途中で何かが壊れます。

このオーバーヘッド、つまりリポジトリとの間のコンテンツの絶え間ない往復は、今日のローカリゼーションツールを使用するすべてのチームが支払う「税金」です。金曜日の午後6時に、翻訳のPRがサイトのビルドを壊した理由をデバッグする立場になるまでは、些細なことに聞こえるかもしれません。

## インターネット以前から受け継がれた設計

ほとんどのローカリゼーションプラットフォームは、現代の開発ワークフローよりも前の概念に基づいて設計されています。翻訳メモリ。ファジーマッチング。独自のエディタ内で作業する人間翻訳者が、データベースから類似の文字列を提案するツールに支えられています。

これらのアイデアは、翻訳が手動でオフラインのプロセスであった時代には理にかなっていました。しかし、企業は翻訳メモリをロックインの仕組みに変えました。あなたの過去の翻訳、つまりあなたが費用を払って得た組織的知識は、彼らのプラットフォーム内に存在します。別のプロバイダーに移行するということは、最初からやり直すか、機能しないエクスポートに費用を支払うことを意味します。

その結果、人工的な摩擦に基づいて構築された業界が生まれました。あなたのコンテンツはリポジリを離れ、ブラックボックスに入り、他者のスケジュールで戻ってきます。

## 誰も語らないCIの問題

[Tuist](https://tuist.dev)では、ドキュメントサイトとマーケティングサイトをローカライズしています。外部のローカリゼーションプラットフォームを使用していた際、それらのツールが開いたプルリクエストが自動チェックに失敗することが頻繁にありました。

問題は構造的です。外部のローカリゼーションツールは、あなたのCIパイプラインを実行できません。それらは、あなたのリンター、ビルドステップ、リンクチェッカー、フロントマターのスキーマを知りません。翻訳されたコンテンツをリポジトリにプッシュし、あとはうまくいくことを願うだけです。

これは本当に苛立たしいものでした。大量の翻訳を受け取ると、PRはCIに失敗し、チームの誰かが作業を中断して、翻訳ツールが引き起こした書式設定の問題、構文エラー、無効なマークアップを修正しなければなりませんでした。それは本来問題になるべきではないことを解決するために、本来の作業からエネルギーを奪いました。

## すべてを変えた対話

l10nのアイデアは、ローカリゼーションにおける彼女の経験がアイデアを結びつける上で非常に重要だった[Maria Jose](https://github.com/mjsesalm)との考察と対話から生まれました。彼女は言語学的側面とツールが抱えるフラストレーションの両方を深く理解していました。

私たちは共に、シンプルな問いを投げかけました。「**ローカリゼーションエージェントが開発者と同じツールを使えたらどうなるだろう？**」

ファイルを同期させるプラットフォームではありません。コードベースから切り離されたWebエディタでもありません。そうではなく、プロジェクト、環境、そして検証ツールにアクセスできるコーディングエージェントです。ファイルを翻訳し、リンターを実行し、出力をチェックし、何かが壊れた場合はプルリクエストを開く前に再試行できるエージェントです。

そのアイデアは、開発が行われる場所にローカリゼーションを近づけ、仲介者を排除することでした。あなたのプロジェクト、環境、そして作業をオーケストレーションするCLIだけです。

## l10nはソフトウェア業界への私たちの贈り物です

私たちがl10nを構築したのは、より多くのソフトウェアがローカライズされることを望んでいるからです。

複雑なプロセスと高価なプラットフォームは、小規模チーム、インディー開発者、オープンソースプロジェクトにとってローカリゼーションを手の届かないものにします。もし翻訳ワークフローに調達プロセス、単語ごとの価格交渉、そして引継ぎを調整するプロジェクトマネージャーが必要となるなら、ほとんどのチームは英語でリリースして終わりにするでしょう。

l10nはオープンソースです。あなたが既にアクセスできるモデルを使用します。ターミナルで実行されます。そして、私たちのツールではなく、あなたのツールで出力を検証します。

ローカリゼーションは、テストスイートを実行するのと同じくらい自然であるべきだと私たちは考えます。

## 次のステップ：人間による入力体験

現在、l10nは開発者向けに設計されたCLIツールです。しかし、ローカリゼーションの品質に貢献するすべての人々が開発者であるわけではないことを私たちは知っています。

OpenAIがCodexのデスクトップアプリでターミナルを抽象化したのと同じように、人間による入力体験がどのようなものになり得るかを探求したいと考えています。言語学者がブランチ、コンパイル、JSONについて考える必要がない体験です。彼らはコンテンツ、コンテキスト、翻訳を並べて見ることができます。修正が必要な箇所を修正します。そして、ツールが残りのすべてを処理します。コミット、検証、プルリクエストの開設などです。

ターミナルは、最も困難な問題が最初に解決される場所であるため、正しい出発点です。しかし、ゴールは、ソフトウェアがあらゆる言語を話せるようにすることに関心のあるすべての人々を歓迎する体験です。

## はじめに

[mise](https://mise.jdx.dev)でCLIをインストールします。

```bash
mise use github:tuist/l10n
```

次に、プロジェクトを初期化します。

```bash
l10n init
```

これにより、リポジトリのルートに`L10N.md`ファイルが作成され、LLMプロバイダー、ソースファイル、ターゲット言語を設定できます。設定が完了したら、次のコマンドでコンテンツを翻訳します。

```bash
l10n translate
```

`l10n status`で期限切れのものを確認し、`l10n check`で出力を検証します。これだけです。登録するプラットフォームも、管理するダッシュボードも、交渉するベンダーもありません。

l10nは完全に無料でオープンソースです。洗練された製品のように感じられるなら、それは意図的なものです。[Tuist](https://tuist.dev)では、オープンソースの作業も製品として扱っています。私たちがリリースするすべてのものと同じ配慮、開発者体験への同じ注意、そして同じ品質基準をl10nにも注いでいます。オープンソースはそのレベルのクラフトマンシップに値すると信じており、それに依存するチームも同様です。

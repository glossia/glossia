如果您的軟體曾以多種語言發布，您就會知道其中的流程。您會選擇一個本地化平台，將其連接到您的代碼庫，然後將剩餘時間都花在管理同步上。內容發送出去，翻譯結果返回，而在此過程中總會出問題。

這種開銷，即內容在代碼庫之間不斷往返的過程，是每個團隊為使用當今本地化工具所付出的代價。這聽起來似乎微不足道，直到您在週五晚上六點，因為一個翻譯 PR 破壞了您的網站構建而不得不進行調試時，才會深有體會。

## 互聯網時代之前的設計遺留

大多數本地化平台的設計理念都早於現代開發工作流程。翻譯記憶庫、模糊匹配、人工翻譯員在專有編輯器中工作，並由從資料庫中建議相似字符串的工具輔助。

這些理念在翻譯是手動、離線的過程時確實有道理。但公司卻將翻譯記憶庫轉變為一種鎖定機制。您過去的翻譯，也就是您付出成本所獲得的機構知識，都存在於他們的平台中。轉移到其他供應商意味著從頭開始，或者為一個從未真正奏效的導出功能付費。

結果就是，這個產業建立在人為的摩擦之上。您的內容離開您的代碼庫，進入一個黑箱作業，然後在別人的時間表上返回。

## 無人談論的 CI 問題

在 [Tuist](https://tuist.dev)，我們將我們的文檔網站和營銷網站進行了本地化。我們使用過外部本地化平台，並且經常發現自己需要處理那些工具發起的、但未能通過我們自動化檢查的拉取請求 (PR)。

這是一個結構性問題：外部本地化工具無法運行您的 CI 流水線。它們不了解您的程式碼風格檢查工具 (linters)、您的構建步驟、您的連結檢查器、您的前置資料模式 (frontmatter schema)。它們只是將翻譯內容推回您的代碼庫，然後聽天由命。

這真的令人沮喪。我們會收到一批翻譯，PR 會導致 CI 失敗，團隊中就有人必須停下手頭的工作，去修復翻譯工具引入的格式問題、錯誤的語法或無效的標記。這耗費了寶貴的精力去解決本不該出現的問題。

## 一場改變一切的對話

l10n 的想法源於與 [Maria Jose](https://github.com/mjsesalm) 的思考和交流，她在本地化方面的經驗對於我們的想法成形至關重要。她對語言學層面和工具帶來的挫折都有深刻的理解。

我們共同提出了一個簡單的問題：**如果本地化代理能夠使用開發者所用的相同工具呢？**

這不是一個來回同步文件的平台，也不是一個與您的代碼庫脫節的網絡編輯器。相反，它是一群編碼代理，可以訪問您的專案、您的環境和您的驗證工具。這些代理可以在打開拉取請求之前，翻譯文件、運行您的程式碼風格檢查工具、檢查輸出，如果出錯則重試。

這個想法是將本地化更貼近開發發生的地點，消除中間環節。僅僅是您的專案、一個環境，以及一個協調工作的 CLI。

## l10n 是我們送給軟體產業的禮物

我們建立 l10n 是因為我們希望有更多軟體被本地化，而不是更少。

複雜的流程和昂貴的平台讓本地化難以觸及小型團隊、獨立開發者和開源專案。如果您的翻譯工作流程需要採購流程、按字計費的價格談判，以及一個專案經理來協調交接，那麼大多數團隊只會發布英文版然後就收工了。

l10n 是開源的。它使用您已經可以訪問的模型。它在您的終端中運行。並且它使用您自己的工具而不是我們的工具來驗證輸出。

我們認為本地化應該像運行您的測試套件一樣自然。

## 下一步：人工輸入體驗

今天，l10n 是一個為開發者設計的 CLI 工具。但我們知道，並非所有為本地化質量做出貢獻的人都是開發者。

我們想探索人工輸入體驗會是什麼樣子，就像 OpenAI 使用桌面應用程式為 Codex 抽象化了終端界面一樣。在這種體驗中，語言學家不需要考慮分支、編譯或 JSON。他們可以並排查看內容、上下文和翻譯。他們精煉需要精煉的部分。而工具則處理所有其他事情：提交、驗證、打開拉取請求。

終端是正確的起點，因為最難的問題通常首先在這裡解決。但我們的終點是提供一個歡迎所有關心讓軟體說出各種語言的人的體驗。

## 開始使用

使用 [mise](https://mise.jdx.dev) 安裝 CLI：

```bash
mise use github:tuist/l10n
```

然後初始化您的專案：

```bash
l10n init
```

這會在您代碼庫的根目錄創建一個 `L10N.md` 文件，您可以在其中配置您的 LLM 提供者、源文件和目標語言。配置完成後，使用以下命令翻譯您的內容：

```bash
l10n translate
```

使用 `l10n status` 檢查哪些內容已過時，並使用 `l10n check` 驗證輸出。就這樣。無需註冊平台，無需管理儀表板，無需與供應商談判。

l10n 完全免費且開源。如果它感覺像一個精緻的產品，那是我們的本意。在 [Tuist](https://tuist.dev)，我們也將我們的開源工作視為產品。我們將相同的關懷、對開發者體驗的相同關注以及相同的品質標準應用到 l10n 中，就像我們對待我們發布的所有東西一樣。我們相信開源值得那樣的匠心，依賴它的團隊也值得。

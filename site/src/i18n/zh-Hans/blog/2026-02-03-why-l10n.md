如果你曾发布过支持多种语言的软件，那对这个流程一定不陌生。你选择一个本地化平台，将其连接到你的代码库，然后把剩余时间都花在管理同步上。内容发出，译文返回，其间总会出些岔子。

这种开销，即内容在代码库内外不断往返的同步成本，是每个团队为使用当今本地化工具而付出的代价。这听起来微不足道，直到某个周五晚上六点，你需要调试为什么一个翻译 PR 破坏了你的网站构建时，你才会深有体会。

## 互联网时代之前遗留的设计

大多数本地化平台的设计理念，都源于现代开发工作流之前。翻译记忆库。模糊匹配。人工译者在专有编辑器中工作，借助数据库中相似字符串的建议工具。

当翻译是一个手动、离线过程时，这些理念是合理的。但公司将翻译记忆库变成了锁定机制。你过去的翻译，你为之付费的机构知识，都存储在他们的平台内。转用其他提供商意味着从头开始，或者付费导出那些永远无法完美工作的数据。

结果是一个建立在人为摩擦之上的行业。你的内容离开你的代码库，进入一个黑箱，然后在别人的时间表上返回。

## 没人谈论的 CI 问题

在 [Tuist](https://tuist.dev)，我们本地化了我们的文档网站和营销网站。我们曾使用外部本地化平台，并经常发现这些工具发起的拉取请求（PR）未能通过我们的自动化检查。

问题是结构性的：外部本地化工具无法运行你的 CI 流程。它们不知道你的 linter、构建步骤、链接检查器、frontmatter 架构。它们只是将翻译内容推回到你的代码库，然后听天由命。

这确实令人沮丧。我们会收到一批翻译，PR 却未能通过 CI，团队中有人不得不停下手中的工作，去修复翻译工具引入的格式问题、语法错误或无效标记。这消耗了本应用于实际工作的时间，去解决那些根本不该出现的问题。

## 一次改变一切的对话

l10n 的想法源于与 [Maria Jose](https://github.com/mjsesalm) 的思考和交流，她在本地化方面的经验对点燃这些想法至关重要。她对语言学方面和工具痛点都有深刻理解。

我们一起提出了一个简单的问题：**如果本地化代理能够使用开发者所用的相同工具呢？**

不是一个来回同步文件的平台。也不是一个与你的代码库脱节的网页编辑器。而是一个能够访问你的项目、你的环境和你的验证工具的编码代理。这些代理可以在发起拉取请求之前，翻译文件、运行你的 linter、检查输出，并在出现问题时重试。

这个想法旨在将本地化更紧密地融入开发工作流，消除中间环节。只需你的项目、一个环境和一个协调工作的 CLI 工具。

## l10n 是我们赠予软件行业的礼物

我们构建 l10n 是因为我们希望更多软件实现本地化，而不是更少。

复杂的流程和昂贵的平台让小型团队、独立开发者和开源项目难以进行本地化。如果你的翻译工作流需要采购流程、按字数计价的谈判以及项目经理协调交付，大多数团队只会选择发布英文版了事。

l10n 是开源的。它使用你已有的模型。它在你的终端中运行。并且它使用你自己的工具验证输出，而非我们的。

我们认为本地化应该像运行测试套件一样自然。

## 接下来：人类输入体验

如今，l10n 是一个专为开发者设计的 CLI 工具。但我们知道，并非所有为本地化质量做贡献的人都是开发者。

我们希望探索一种人类输入体验，就像 OpenAI 为 Codex 推出桌面应用，将终端抽象化一样。在这种体验中，语言专家无需考虑分支、编译或 JSON。他们只需并排查看内容、上下文和译文。他们完善需要完善的部分。而工具则处理所有其他事情：提交、验证、发起拉取请求。

终端是正确的起点，因为它能首先解决最棘手的问题。但终点是一个能让所有关心软件能讲各种语言的人都能参与的体验。

## 开始使用

使用 [mise](https://mise.jdx.dev) 安装 CLI：

```bash
mise use github:tuist/l10n
```

然后初始化你的项目：

```bash
l10n init
```

这会在你的代码库根目录创建一个 `L10N.md` 文件，你可以在其中配置你的 LLM 提供商、源文件和目标语言。配置完成后，使用以下命令翻译你的内容：

```bash
l10n translate
```

使用 `l10n status` 检查哪些内容已过期，并使用 `l10n check` 验证输出。就是这么简单。无需注册平台，无需管理仪表盘，无需与供应商谈判。

l10n 完全免费且开源。如果它感觉像一个打磨完善的产品，那是有意为之。在 [Tuist](https://tuist.dev)，我们也把开源工作视为产品。我们投入了与发布其他产品相同的关怀、对开发者体验的关注以及相同的质量标准到 l10n 中。我们相信开源值得这种精湛的工艺，依赖它的团队也同样值得。

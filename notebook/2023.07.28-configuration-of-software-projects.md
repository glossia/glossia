# Configuration of software projects

This document captures some decisions around how software projects are configured to integrate with Glossia.

## Format considerations

While some tools in the ecosystem, such as [Babel](https://babeljs.io/docs/configuration), choose to offer a high degree of flexibility, we've recognized that this can lead to decision paralysis and hinder the user experience, especially for non-developers unfamiliar with the assortment of common formats. As such, we have consciously **decided against supporting multiple formats**.

We also strongly believe in the importance of a seamless editing experience that includes auto-completion, validation, and accessible documentation. Without such integrated functionality, developers often find themselves shuffling between documentation and the editor, or relying on the platform to validate formats after pushing changes upstream. Our commitment to an optimal user experience informs our choice of JSON as the most fitting format.

Notably, JSON and its defining standard, JSON Schema, are extensively supported by numerous editors, obviating the need for users to install additional tools in their environment. Online editors, including GitHub, provide robust support for JSON as well.

In order to accommodate user comments, we have opted for an enhanced version of JSON that supports commenting. This modification helps ensure our users not only benefit from JSON's established advantages but also have the added ability to make notes and clarifications directly within the code.

## Format

Some tools in the ecosystem decide to provide a high level of flexibility here (e.g. Babel), however, we believe having many options to choose from might cause some paralysis and friction for people coming to the platform, specially considering that some people that might configure the project are not developer and therefore might not be familiar with any of the common formats at all. Therefore, we discarded the idea of supporting multiple formats.

Moreover, we think it's important that when a person edits the file, the editor can asist in the editing providing auto-completion, validation, and documentation. Without an integrated editing experience, developers have to jump back and forth between the documentation and the editor and very often end up pushing the changes upstream for the platform to validate the format. We want our users to have the best editing experience, and that makes JSON the most suitable candidate.

First and foremost, the format and its standard for defining schemas, JSON Schema, is widely supported by many editors out there. In other words, developers don't have to install additional tools in their environment. Online editors like GitHub's provides a great support for it as well.

Because we'd like users to be able to add comments, we are using an enhanced version of JSON that supports comments.

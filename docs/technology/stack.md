# Stack

When we began developing Glossia, **cost-effectiveness** was paramount to successfully bootstrap our business. Consequently, our technological choices were geared towards **optimizing both financial expenditures and [reducing complexity](https://en.wikipedia.org/wiki/Programming_complexity)**. This focus allowed us to concentrate solely on Glossia's growth, eliminating distractions like scaling services or addressing test suite inconsistencies.\
At the time, **JavaScript** and **TypeScript** were dominant programming languages, bolstered by a thriving ecosystem that enticed many organizations to incorporate them into their tech stack. Yet, as I've articulated in [this blog post](https://pepicrft.me/blog/2023/08/05/abstracted-complexity), JavaScript often becomes a maze of abstractions that can inadvertently lead to bugs, hindered developer workflows, or sluggishness. It's an inherent trait of this ecosystem. While many developers relish delving into its intricacies, choosing this path would have diverted our attention from building Glossia. Although it might have expedited our project's initiation, we **didn't see it as a sustainable long-term choice**. Many companies, having initially embraced frameworks like [NextJS](https://nextjs.org/), later lament their decisions, especially when confronted with slow compilations and limited control.

We ultimately chose to construct a monolithic service using [Elixir](https://elixir-lang.org/) and [Phoenix](https://www.phoenixframework.org/). In the subsequent sections, I'll delve into the reasoning that informed our decision and discuss the compromises we accepted along the way.

## Elixir

When I was in the early stages of developing Glossia, I had already been experimenting with Elixir for several months. My initial foray into the language was sparked by a podcast interview with its creator, [José Valim](https://github.com/josevalim). To provide some background, José, the creator of Elixir, had a rich history as a Ruby developer and was a core member of the Ruby on Rails team. His quest was to **find a programming language capable of leveraging all available computer cores without the inherent risks of memory issues**, a common challenge in languages supporting parallelism across multiple processors. His search led him to [Erlang](https://www.erlang.org/) and its [Virtual Machine (VM)](https://en.wikipedia.org/wiki/BEAM_(Erlang_virtual_machine)). However, José desired to combine the delightful syntax of Ruby with a comprehensive tool ecosystem. This ambition drove him to craft a new language altogether: **Elixir**.\
*What sets Elixir and Erlang apart?* Their virtual machine is akin to an operating system. In this environment, a program consists of [lightweight processes](https://www.erlang.org/doc/reference_manual/processes.html), each with its own memory and CPU. These processes communicate with others through message passing and can be monitored. At first glance, this might seem unremarkable. However, conceptually, it wields significant power, especially for projects like Glossia. By using processes as error boundaries and supervisors for self-recovery, we can build **inherently fault-tolerant systems**. And let's face it, errors are inevitable due to unpredictable external factors that software cannot control.

The VM adeptly schedules these events across all available processors in the system. Thanks to the isolated memory and CPU of each process, **data races are non-existent**. This ability to enhance performance simply by allocating more CPU cores is consequential in several ways. In a production setting, our initial scaling strategy is vertical: *we can add more cores to our servers*. The counterpart to this is horizontal scaling, which tends to introduce complexity in production. This complexity arises from the necessity of orchestrating deployments using advanced tools like Kubernetes and employing reverse proxies to route traffic to the appropriate instance. While we might eventually resort to horizontal scaling for reasons such as geographic availability, our strategy allows us to postpone this decision. Another significant, yet often overlooked, advantage is in testing. **If our test execution ever lags, we can employ the same multicore approach to expedite it.**

Elixir's **functional nature** ensures that both business logic and tests are deterministic, making them straightforward to comprehend. This intrinsic quality has profound implications, especially in testing. Due to the **deterministic nature and absence of global state mutation**, tests can be run in parallel without introducing unpredictability or "flakiness". In contrast, in non-functional languages where state mutation is prevalent and cannot always be curtailed, test flakiness often surfaces, especially when the sequence of test execution changes or tests run concurrently. I've observed entire teams of up to five individuals working full-time solely to monitor and manage such test flakiness and execution times. *Isn't it remarkable that Elixir sidesteps this issue entirely?*

Several other factors also influenced our choice. The ease with which the system can be scaled horizontally is noteworthy. This allows us to potentially [deploy dedicated instances for GPU-intensive tasks](https://github.com/elixir-nx/bumblebee), such as AI. Furthermore, Elixir's thriving community and integrated toolchain amplify its appeal. By adopting Elixir, we're in good company, alongside successful firms like [Fly](https://fly.io) and [Discord](https://discord.com) who have already championed the technology.

Of course, we recognized potential **challenges**, notably in recruitment. It might be harder to find candidates familiar with Elixir or those keen to learn it. However, our strategy emphasizes hiring versatile problem solvers and software engineers who view technology as a means to an end rather than the end itself. We're investing in a robust architecture and an engaging development environment, ensuring that newcomers find it intuitive and enjoyable to navigate through Elixir code.

## Deno

Glossia relies on transient Linux environments for specific tasks. For instance, when new commits are added to a repository, Glossia accesses this repository to parse configuration files and determine glob patterns, creating a localization request payload. For these operations, we leverage cloud-based services like [Google Cloud's "Cloud Build."](https://cloud.google.com/build) Such services necessitate a Docker image packed with the necessary software and accompanying execution commands, unless one opts against the default.

Had we implemented this logic in Elixir, we would face the tedious routine of constantly building and uploading images to a registry for every change, especially when assessing them in environments mirroring production. This isn't the streamlined experience we envision for developers. An alternative might have been embedding a script within the Docker image. This script, upon receiving a commit sha, would retrieve the source code and its dependencies for execution. Yet, this means more maintenance on our end. Instead, **why not leverage a runtime that offers this feature set by default**, coupled with an exceptional developer experience? Enter: [Deno](https://deno.com/).

Deno excels at dynamically resolving a decentralized ESM dependency graph over the internet. *The immediate benefit?* The runtime natively handles code and dependency retrieval. For us, this translates to merely serving the module graph from our server with appropriate content-type headers, and Deno handles the retrieval and execution. This approach ensures the builder process (our term for it) utilizes the server-deployed version. Furthermore, Deno's focus on security becomes invaluable as we aim to expose more developer APIs in the future. For instance, as we pave the way for project onboarding through the builder, developers will have dedicated APIs guiding Glossia on integrating diverse technologies.

**Lastly, Deno's developer experience is unparalleled.** Gone are the days of juggling multiple NPM packages for fundamental operations. With Deno, testing, linting, formatting, and compilation are integrated features, devoid of external tools. It's a refreshing change, especially for those accustomed to Node's intricacies.
